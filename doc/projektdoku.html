1 Einleitung

Back to top

In der Schweiz bieten viele Bibliotheken neben dem klassischen Buchverleih weitere Zusatzdienstleistung wie das Veranstalten von eigenen Events an. Diese Veranstaltungen werden von den Bibliotheken auf ihren eigenen Websites angeboten, einen bibliotheksübergreifenden Veranstaltungskalender gibt es jedoch nicht. Dies soll durch ein Praxisprojekt im Masterstudiengang Information and Data Management an der HTW Chur geändert werden.

Der Auftraggeber Prof. Bruno Wenk hat hierfür ein Konzept zur Realisierung eines zentralen Veranstaltungskalenders erarbeitet. Das Konzept fusst auf den Grundsätzen des „Semantic Web“, bei dem Anbieter von Daten (Bibliotheken) diese mit Metainformation anreichern und als maschinenlesbares Format im Internet bereitstellen. Daraufhin können automatisierte „Web-Crawler“ diese Daten (zentraler Veranstaltungskalender) regelmässig abgreifen und weiterverarbeiten. Dateiformat und Schema wurden auf JSON-LD Format nach schema.org Definition festgelegt. Die Erstellung der JSON-LD Daten soll dabei für das Bibliothekspersonal möglichst einfach sein.



In den nachfolgenden Abschnitten werden Auftrag und Ziele genauer definiert, die gewählte Vorgehensweise beschrieben sowie die festgelegten Phasen im Projektverlauf erläutert.

Die Use Cases im zweiten Kapitel sollen aufzeigen, welche Prozesse hinter dem Tool stehen.

Das dritte Kapitel befasst sich mit den Komponenten, welche das Tool beinhalten soll, im vierten Kapitel werden dann die genutzten Technologien für die Prozessschritte festgehalten und erklärt.

Kapitel fünft zeigt den Stand des Tools bei Abschluss des Projektes auf und bietet damit einen Überblick, was konkret vorhanden ist.

Bevor im siebten Kapitel das Projekt reflektiert und ein Blick in die Zukunft geworfen wird, sollen bereits in Kapitel sechst erste Handlungsempfehlungen und Ideen für die Weiterentwicklung des Tools konkretisiert werden.

1.1 Auftrag und Zielsetzung

Ausgangslage für das Projekt waren der schriftliche Auftrag, die Auftragsbesprechung mit dem Referenten sowie weitere Projekte um die Seite http://mas-is.lima-city.de/, welche in anderen Kursen durchgeführt wurden.

Die Projektgruppe erhielt den Auftrag, ein webbasiertes Tool zu entwickeln, welches auch Bibliotheksangestellten ohne spezifische Kenntnisse von JSON-LD die Erfassung von Events ermöglicht. Das Tool soll mittels XML-File konfigurierbar sein, etwa um im Vorfeld definieren zu können, welche Mindestangaben zu den jeweiligen Events erfasst werden sollen. Die eingegebenen Daten sollen dann auf Knopfdruck in das erforderte JSON-LD Format umgewandelt werden und für die weitere Bearbeitung durch die Website-Administration bereitgestellt werden. Die Administration kann die JSON-LD Daten auf der Website der Bibliothek veröffentlichen und diese so semantisch auszeichnen. Eine zweite, separate Projektgruppe ist verantwortlich, den Veranstaltungskalender zu entwickeln und die online gestellten JSON-LD Daten jeder Bibliothek abzugreifen.

3 Komponenten des Tools

Back to top

Nach ersten technischen Entwicklungsschritten und vor Überlegungen zum Design wurden die einzelnen Elemente des Projektes aufgegriffen und unterteilt in Endnutzer (EN) wie den Bibliotheksmitarbeitenden und Professionals bzw. Profis (PRO), welche das Tool weiterentwickeln könnten. Entstanden ist eine Unterteilung in drei Hauptkomponenten (s. Anhang A - Komponenten des Tools), welche nachfolgend erläutert werden.

3.1 Formular

Kern des Projektes ist ein Formular, das in erster Linie Veranstaltungen von Bibliotheken abdeckt. Nach Ausfüllen durch den Nutzer soll per Knopfdruck ein JSON-LD-Datensatz generiert werden, welcher dann in die Website eingebunden werden kann.

Mit der Einbindung des JSON-LD-Datensatzes wird die Website hinsichtlich der Veranstaltungstermine semantisch so ausgezeichnet, dass der vorgesehene Webcrawler die nötigen Daten identifizieren und in strukturierter Form abholen kann. Ein Nebeneffekt ist, dass dadurch die Webseiten der Bibliotheken für Suchmaschinen optimiert werden, sofern die JSON-LD Tags gemäss den Richtlinien der Anbieter erstellt worden sind (siehe Abschnitt 4.2.6 JSON-LD). Das Formular bietet für Nutzer ausserdem den Vorteil, dass sie keine Kenntnisse von schema.org oder JSON-LD benötigen, sondern lediglich die Webadministratoren kennen oder selber mit ihrem CMS und der Einbindung von script-Tags in den <head>-Teil der Website vertraut sein sollten.

3.2 Tutorial

Als Unterstützung für die Endnutzer wird ein Tutorial zur Verfügung gestellt. Dieses erläutert dem Nutzer schrittweise, wie das Formular auszufüllen ist. Ebenso wird kurz erläutert, wie der JSON-LD-Datensatz in die Website integriert werden kann.

Da die Formulare in den Sprachen Deutsch, Englisch, Französisch und Italienisch zur Verfügung stehen, wird auch das Tutorial übersetzt. Dadurch wird Nutzern aus allen Regionen der Schweiz und darüber hinaus die Anwendung des Tools erleichtert.

3.3 Entwickler

Ein Grundgedanke des Projektes ist die Weiterentwicklung des Tools, weshalb dies der umfassendste Bereich darstellt. Bei der Entwicklung wird unterschieden zwischen der Erstellung neuer Formulare und der Erweiterung der Plattform. Im Entwickler-Bereich finden sich Anleitungen und Anweisungen für die Formulare, der Source Code für das Tool (beschränkt auf JS und XSLT) sowie die hier vorliegende Projektdokumentation mit den technischen Details, welche im nächsten Kapitel 4 Design und technische Details folgen. Weitere Kommentare zum Source Code finden sich direkt in den Files (beispielsweise PHP).

3.3.1 XML-Anleitung und Validator

Damit zukünftig neue Formulare erstellt werden können, wird ein Tutorial für Entwickler angeboten. In dieser Anleitung wird aufgezeigt:

  • wie das XML-File für ein neues Formular aufgebaut sein muss
  • welche Elemente und Attribute Verwendung finden (obligatorisch oder optional)
  • wo die Datei auf dem Server abgelegt werden muss
  • wie das File validiert werden kann

Für die Validierung wird ein eigener Bereich zur Verfügung gestellt. Die Grundidee der vorgängigen Validierung der Files ist die Sicherstellung, dass ein XML-File korrekt für das Bibliothekspersonal aufbereitet wird und anschliessend ohne Komplikationen genutzt werden kann.

Die Verfasser der XML-Templates laden diese hierfür in das separate Verzeichnis [WEBROOT]/xmltovalidate² auf den Server. Danach können sie auf einfache Art und Weise das hochgeladene XML-File via [meine_URL]/validate validieren. Wenn bei der Validierung keine Fehler entdeckt werden, kann der Entwickler das validierte File auf Knopfdruck "veröffentlichen". Das File wird dabei serverseitig in das Verzeichnis [WEBROOT]/xml kopiert und somit für den eigentlichen JSON-LD Prozess zur Verfügung gestellt.

Das XML-File wird 4-Stufig validiert:

  • 1. Stufe - Fehler

    Prüfung ob das XML-File den Anforderungen gemäss XSD entspricht. Erfolgt serverseitig via PHP "default" XML-XSD Validierungsfunktion.

  • 2. Stufe - Fehler

    Prüfung nach Logik-Fehlern wie beispielsweise Check, dass bei als sichtbar konfigurierten Labels auch ein Wert für die Labels mitgegeben wird. Erfolgt serverseitig via XPath-Abfragen.

  • 3. Stufe - Warnung

    Es wird eine Warnung ausgegeben, falls nicht alle Elemente eine @type-Annotation besitzen.

  • 4. Stufe - Info

    Ein Info-Text wird ausgegeben, falls nicht alle Elemente eine @id-Annotation besitzen.

Wie bei Validierungen üblich, gibt es verschiedene Schweregrade.

Stufe 1 und 2 haben den Schweregrad Fehler (Error). Das XML-File kann dann nicht für das Bibliothekspersonal zur Verfügung gestellt werden.

Stufe 3 hat den Schweregrad Warnung (Warning). Dabei wird der validierenden Person nahegelegt, eine Korrektur vorzunehmen, das XML-File kann aber trotzdem verfügbar gemacht werden.

Stufe 4 hat den Schweregrad Info. Es wird lediglich eine Zusatzinfo angezeigt, wie sich das XML-File noch verbessern liesse, das XML-File kann aber ohne weiteres publiziert werden.

Jede angezeigte Meldung enthält eine Zusammenfassung des Fehlers und die entsprechende Zeilennummer im XML-File.

3.3.2 Source Code-Dokumentation (JavaScript und XSLT)

Die Dokumentation des Source Codes umfasst sämtliche Files aus dem Code-Ordner „js“ und beschränkt sich dadurch auf JavaScript und XSLT. Grundlage für die Zusammenstellung waren die Kommentare im Source Code, für die Erstellung wurde JSDoc/XSLTDoc zu Hilfe gezogen.

JSDoc ist eine Programmschnittstelle (auch API genannt) für JavaScript, mit der aus Programmcode automatisch eine Dokumentation generiert werden kann.

Durch klares Kennzeichnen des zu dokumentierenden Codes mittels /** und der Vergabe von Tags wird dem JSDoc-Parser ermöglicht, Kommentare im Quellcode zu erkennen, zuzuordnen und zusammengefasst in einer eigenen Website im HTML-Format wiederzugeben (JSDoc 3 2017).

Die Sequenzen für die Source Code Dokumentation wurden vor den entsprechenden JavaScript-Functions definiert. Weitere erläuternde Kommentare im Source Code (auch in anderen Code-Files wie z.B. PHP) wurden auskommentiert und werden nicht separat in der Dokumentation aufgeführt.

Bei der Dokumentation des Source Codes wurde der Open-Source Gedanke berücksichtigt, indem sämtliche Files in Englisch kommentiert wurden.

4.1 Entscheidungen betreffend Design

Das Design für die Repräsentation des Tools wurde ausgehend von den definierten und im vorausgehenden Kapitel erläuterten Komponenten entwickelt. Um diese an einem Ort zu vereinen und damit unter anderem eine einfache Weiterentwicklung zu gewährleisten, wurde entschieden, eine umfassende Website zu erstellen.

Anhand einer ersten Skizze wurde der Aufbau der Seite definiert, der sowohl den Endnutzern bzw. dem Bibliothekspersonal gerecht werden soll, als auch den Entwicklern, welche dieses Projekt weiterführen möchten.



Um die Nutzung der Formulare für Anwender zu erleichtern, wird das Portal aufgeteilt in einen zentralen Anwendungsbereich für Endbenutzer sowie in einen abgegrenzten und klar erkennbaren Entwicklerbereich, in der Skizze rot und grün gekennzeichnet. Der Bereich für die Endnutzer beinhaltet die beiden roten Menüpunkte und umfasst einerseits die Seite mit den Formularen sowie andererseits die Seite mit dem Tutorial. Dadurch wird gewährleistet, dass die Formulare schnell auffindbar sind und ohne Umwege genutzt werden können, die Nutzer aber dennoch über die Existenz eines Tutorials Bescheid wissen und bei Problemen schnell darauf zugreifen können.

Der Entwicklerbereich ist für den Endnutzer nicht relevant und kann problemlos ignoriert werden. Für die Verantwortlichen, die sich mit der Weiterentwicklung des Projektes beschäftigten, soll jedoch eine gute Basis zur Verfügung gestellt werden. Der Bereich wiederum ist aufgeteilt in drei Unterbereiche, welche bereits in Abschnitt 3.3 Entwickler genauer definiert wurden.

Für die Umsetzung des Frontend der Website und die Darstellung der Elemente wurde auf Bootstrap 3 gesetzt. Die aktuelle Alpha-Version von Bootstrap 4 (v4.0.0-alpha.6) wurde aufgrund von Kompatibilitätsproblemen (insbesondere mit IE11) abgetan.

4.2 Verwendete Technologien

In diesem Abschnitt werden die genutzten Technologien und Standards erläutert und im Kontext des Prozesses dargestellt.

4.2.1 XML

Ausgangslage für das Formular ist ein XML-File, das von Experten erstellt und eingepflegt wird. Das File definiert, welche Inhalte via #wilhelmtell von dem Bibliothekspersonal erfasst werden sollen.

Extensible Markup Language, kurz XML, ist eine vom W3C³ definierte Auszeichnungs- und Metasprache. XML bestimmt, wie „benutzerspezifische Vokabeln in XML-Dokumenten zu verwenden sind“. Diese Vokabeln (sogenannten „tags“, gekennzeichnet mit <…>) wiederum beschreiben die Textabschnitte, die sich in diesem XML-Dokument befinden (Moos 2008, S. 1).

Wie ein XML-Template für das Tool #wilhelmtell aufgebaut sein muss, wird im dafür angefertigten Tutorial erläutert, beschrieben in Abschnitt 3.3.1 XML-Anleitung und Validator.

Als (technische) Design-Alternative zur Generierung von HTML-Formularen kann die Auszeichnungssprache XForms verwendet werden, welche stark mit XML verbunden ist. Der W3C-Standard stellt eine XML-Spezifikation für webbasierte Formulare dar und war als Nachfolger von HTML-Formularen angedacht. XForms trennt im Gegensatz zu HTML die Darstellung des Formulars von der Verarbeitung der Daten und orientiert sich damit am Model-View-Controller-Prinzip (W3C 2009)4.

4.2.2 XSD

Das vorgängig definierte XML-File wird auf den Server, in den Ordner [WEBROOT]/xmltovalidate hochgeladen und kann danach maschinell mittels XML-Schema (XSD) Validierung und XPath Abfragen validiert werden.

Die W3C-Empfehlung XSD definiert die Strukturen von XML-Dokumenten und gibt Elemente und Attribute vor, die verwendet werden dürfen. Diese Vorgaben wiederum ermöglichen eine Validierung bzw. Überprüfung von XML-Files auf deren Wohlgeformtheit5. Erst eine erfolgreiche Validierung macht das XML-File für eine Anwendung verständlich und ermöglicht dadurch die (W3C 2004).

Es wird XSD Version 1.0 verwendet: XSD Version 1.1 bietet zwar die Möglichkeit an erweiterte Schema-Regeln Regeln direkt im XSD-File zu definieren (basierend auf XPath 2.0 Abfragen), wird aber generell wenig unterstützt. Auch die vom Projektteam verwendeten Testserver der HTW-Chur unterstützen nur Version 1.0, für die Validierung wurde deshalb ein zweiteiliger Ansatz gewählt:

Die Basis-Validierung erfolgt mittels XML-Schema (XSD v1.0) Verifikation. Im nächsten Schritt werden komplexere Prüfungen, PHP-serverseitig mittels separaten XPath (v1.0) Abfragen durchgeführt. Die XPath Abfragen sind dabei so gestaltet, dass für ein positives Ergebnis ein leeres Resultat erwartet wird. Gibt die Abfrage ein Resultat zurück, so handelt es sich um einen Fehler und das Abfrageergebnis kann direkt an den Benutzer zurückgegeben werden. Beispiel: Suche nach allen Feldern ohne Label  Wenn etwas gefunden, dann „FAIL“ und gebe die jeweilige Zeilennummer der Resultate an, sonst „PASS“.

Wie die Validierung im Detail funktioniert, kann in Abschnitt 3.3.1 XML-Anleitung und Validator nachgelesen werden.

4.2.3 XSLT

XML-Files, welche die Validierung erfolgreich durchlaufen haben und anschliessend publiziert wurden, können von dem User in der Formular-Maske ausgewählt werden. Beim Anwählen, wird das selektierte XML-File in HTML umgewandelt. Diese Umwandlung erfolgt mittels automatisierter XSL Transformation (XSLT).

XSLT ist ebenfalls eine Empfehlung des W3C und definiert eine Sprache, welche XML-Dokumente in ein anderes Dokument umwandelt. Zentrale Aufgabe von XSLT ist es, „Styling-Informationen zu einem XML-Quelldokument hinzuzufügen“6. Dazu kann beispielsweise eine Umwandlung in ein darstellungsorientiertes Format wie SVG, HTML oder XHTML stattfinden. Es gibt jedoch auch weitere Transformationsaufgaben, für die XSLT verwendet wird. „Eine in XSLT ausgedrückte Transformation beschreibt Regeln für die Transformation von Eingangsdaten in Ausgangsdaten“ (W3C 2017).

4.2.4 HTML

Nach erfolgter Transformation steht dem User eine einfache und verständliche Eingabemaske bzw. ein HTML-Formular zur Verfügung. In das Formular können nun alle erforderlichen und relevanten Daten für eine Veranstaltung eingegeben werden.

HTML steht für HyperText Markup Language und beschreibt die gängige Auszeichnungssprache, die zur Erstellung von Websites verwendet wird. Mit HTML können Dokumentinhalte wie z.B. Kopfdaten (header), anzuzeigender Inhalt (body) oder Fusszeilen (footer) sowie Elemente wie Überschriften, Absätze, Grafiken, etc. ausgezeichnet werden. Diese Auszeichnungen werden von Webbrowsern aufgelöst und entsprechend am Bildschirm, für den Nutzer grafisch aufbereitet, ausgegeben (SELFHTML 2017).

Für die gestalterische Bearbeitung des Formulars in #wilhelmtell ist zu berücksichtigen, dass keine HTML-Elemente hinzugefügt werden sollten, die sich statisch auf die IDs von anderen HTML-Elementen beziehen. Da alle Elemente geklont werden können und bei einem solchen Vorgang die IDs neu generiert werden, wird die Referenzierung auf statische IDs erheblich erschwert. Beispielsweise werden explizite label-Elemente, also labels, die auf die ID von input-Elementen referenzieren, nicht unterstützt. Stattdessen wird die indirekte Referenzierung verwendet (input-Element als Kind-Element des labels). Diese Tatsache stellt primär aber kein Problem dar, da gemäss W3C-Standard implizite Referenzierungen vollumfänglich unterstützt werden.

4.2.5 JavaScript

Hat der User alle relevanten Daten eingegeben, kann die Transformation zu JSON-LD durchgeführt werden. Hierzu klickt der User auf den entsprechenden Button (JSON erzeugen), woraufhin die eingepflegten Daten mittels JavaScript in das JSON-LD Format übertragen werden.

JavaScript (JS) ist eine von der Firma Netscape entwickelte Skriptsprache, mit der Funktionen für Interaktionen in eine Website integriert und statische HTML-Websites dadurch dynamisch erweitert werden können. Das JS-Skript wird bei Aufruf der Seite von einem sogenannten Interpreter in Maschinencode umgewandelt und ausgeführt (Bewersdorff 2014, S. 42).

Im Code-Repository gibt es ein JS-File mit Helferklassen, anhand dessen die Attribute aller relevanten Elemente via diese Helferklassen bearbeitet werden können. Zusätzlich kann man in dem JS-File auch die Texte bzw. Error-Messages anpassen. Das JS-File heisst formattingHelper.js, die Nutzung wurde, wo sinnvoll, mit Anwendungsbeispielen verdeutlicht, sollte aber grundsätzlich selbsterklärend sein.

4.2.6 JSON-LD

Der JSON-LD-Datensatz wird dem User in einem separaten Ausgabefeld angezeigt. Diese kann der User abspeichern oder kopieren und den IT-Experten bzw. Administratoren zur Veröffentlichung auf der Website der Bibliothek aushändigen.

JavaScript Object Notation for Linked Data, kurz JSON-LD ist eine Annotation für Daten, die automatisiert verarbeitet werden sollen und bildet damit unter anderem eine Grundlage für die semantische Auszeichnung von Websites. JSON-LD hat gegenüber Mikrodaten, RDFa, etc. einige Vorteile. Einerseits ist ein JSON-LD-Datensatz kompakter und für Menschen leichter lesbar, andererseits ist keine konkrete Unterstützung durch Content Management Systeme erforderlich, da der Datensatz einfach mittels separatem script-Tag in den Header integriert werden kann (Ryte GmbH, o.J.).

Bei der Generierung des JSON-LD Textes im Tool #wilhelmtell ist zu berücksichtigen, dass alle leeren Objekte vor dem Anzeigen gelöscht werden. Das, damit nur relevante Attribute angezeigt werden. Leer definiert sich hier als leeres Array, leeres Objekt oder als Objekt mit nichts ausser einem default @type-Attribut.

Strukturierte Daten wie JSON-LD können unter anderem auch für eine bessere Auffindbarkeit in Suchmaschinen (SEO) wie z.B. Google genutzt werden. Ob der generierte JSON-LD-Datensatz demnach auch für die Google-Crawler auffindbar ist, kann getestet werden via Google Testing Tool.

Das XML-Template (s. Abschnitt 4.2.1 XML) sollte hierfür gemäss Google-Richtlinien7 aufgebaut werden.

4.3 Technische Details zur Anwendungsebene

Die folgenden Abschnitte gehen auf weitere notwendige technische Details ein und erläutern unter anderem, welche Browser das Tool unterstützen, welcher Server benötigt wird und was es in Bezug auf das Formular bzw. einzelne Felder zu berücksichtigen gibt.

4.3.1 Browser-Support

Unterstützt werden mindestens alle gängigen Browser der aktuellen und letzten Version (Stand Ende 2017). Kompatibel sollten jedoch alle Web-Engines sein, die JavaScript ECMAScript 5 unterstützen. Es wird vermutet, dass die native Version des Internet Explorer 9 und solche darunter nicht unterstützt werden, da diese nicht zuverlässig getestet werden konnten. Es wird empfohlen, jede Änderung des Tools insbesondere auf die Kompatibilität der Trident Engine (vor allem MS Internet Explorer – ohne Edge) zu testen.

4.3.2 Serverseitige Tasks (PHP)

Alle serverseitigen Aufgaben erfolgen via PHP Script, weshalb das Tool auf einem PHP-Server laufen muss. PHP Version 5 bis 7 werden unterstützt, unter der Voraussetzung, dass die Standard Library zur XML-Verarbeitung installiert ist (XSD und XPath v1.0). Es werden PHP-Funktionen verwendet, die in den PHP-Versionen 4 und darunter nicht oder nur teilweise unterstützt werden.

Folgende Tasks erfolgen serverseitig:

  • XSD-Validierung
  • Fetching8 der Währungen
  • Fetching und maschinelle Übersetzung von schema.org-Beschreibungen
  • Fetching der XML-Files auf dem Server

Währungs- und Schema.org-Fetch haben einen Fallback auf die lokale Datei im Ordner [WEBROOT]/php/local-bakfalls die remote-Files nicht verfügbar wären. Der Order „local-bak“ ist relevant und sollte nur sporadisch aktualisiert werden. Eine lokale „fallback-Datei“ wird verwendet, da damit das Tool auch ohne Internet-Verbindung funktioniert. Zudem können remote-Abfragen immer mal wieder fehlerhaft sein (beispielsweise remote-host offline oder host-Datei fehlerhaft). Die lokale „fallback-Datei“ könnte auch automatisiert periodisch heruntergeladen werden (beispielsweise via Cron-Job). Bei dieser Variante wird ein manuelles nach-testen strengstens empfohlen.

Auch wurden Tutorial- und Entwickler-Seiten unter PHP entwickelt, da dadurch die mehrsprachige Ausarbeitung sowie die Informationsarchitektur, welche zwei Hierarchieebenen übersteigt, besser gelingen konnten. Durch den modularisierten Aufbau der gesamten Anwendung können Redundanzen im Code vermieden werden, was Fehlerquellen minimiert. Die Seite kann damit leichter und flexibler erweitert werden.

4.3.3 Besonderheiten des Formulars

Das Formular, welches dem Endnutzer angezeigt wird, bringt einige besondere Merkmale mit sich. Diese sind auch bei der Weiterentwicklung zu berücksichtigen, da sie Dependenzen mit der Kernanwendung haben.

  • Validierung der Eingabefelder

    Die Eingabemaske wird auf zwei verschiedene Arten validiert:

    Pflichtfelder werden für alle Browser ausser für MS-IE mittels Standard HTML5 Validierung gemacht. Die Dialoge der HTML5-Validierung sind jeweils immer in der Systemsprache des Users. Das bedeutet, dass bei Systemsprache Deutsch eine Meldung in Deutsch ausgegeben wird, auch wenn für das Formular die Sprache Französisch (FR) ausgewählt wurde. Aufgrund von Kompatibilitätsgründen wurde für die Trident-Engine (beispielsweise MS-IE ohne Edge) eine eigene Validierung mittels Popups entwickelt. Diese Popups sind folglich in der Sprache, die im Formular eingestellt wurde (nicht zwingend die Systemsprache).

    Die Validierung der Datentypen, ob also beispielsweise auch wirklich eine Zahl in ein Zahlenfeld eingegeben wurde, wird mittels regex überprüft, allfällige Fehlermeldungen werden via Bootstrap Elemente angezeigt. Hier ist zu beachten, dass bei Drücken des „JSON-erzeugen“-Buttons geprüft wird, ob noch Fehlermeldungen zum Format bestehen. Sollen die Format-Fehlermeldungen angepasst werden, muss dies auch bei der „finalen“ Prüfung vor der JSON-LD-Generierung mitberücksichtigt werden.

  • Datumsfelder

    Alle Datumsfelder werden live an die Zeitzone des Users angepasst. Wenn im XML-Template beispielsweise der Default-Wert Zeitzone US-Ostküste gespeichert ist, wird das Datum beim Aufrufen in die lokale Zeit konvertiert (lokale Zeit entspricht der Lokalisation des Clients).

  • Add-/Delete-Buttons

    Jedes Node-Element kann mit einem Klick auf den „löschen“-Button gelöscht werden. Pflichtelemente können nur solange gelöscht werden, wie es mehr als ein gleiches Element gibt. Das letzte Pflichtelement kann nicht gelöscht werden. Weitere Node-Elemente können via „hinzufügen“-Button hinzugefügt werden. Bei erstmaligem Aufruf des Formulars werden relevante Elemente geklont, deren Inhalt gelöscht und zwischengespeichert. Bei Klick auf „hinzufügen“ wird schliesslich das zu klonende Element identifiziert und innerhalb der entsprechenden Hierarchiestufe zuunterst hinzugefügt. Das Klonen der Elemente bei erstmaligem Aufruf („Blueprinting“) ermöglicht es, dass ein Element hinzugefügt werden kann, auch wenn keine „peer-Elemente“ mehr existieren (beispielsweise Hinzufügen eines Fotos, auch wenn keine vorgehenden Fotos mehr vorhanden sind).

Das übergreifende Ziel des Projektes ist ein Tool, welches Bibliotheken in der semantischen Auszeichnung von Veranstaltungen auf ihrer Website unterstützen soll. Dieser Schritt ist erforderlich, damit der von Bruno Wenk geplante bibliotheksübergreifende Veranstaltungskalender die Daten auslesen kann. Mit der Website #wilhelmtell wird ein einfaches Tool zur Verfügung gestellt, das zudem Potential für Weiterentwicklungen bietet. Für diesen Auftrag wurden in Abschnitt 1.1 Auftrag und Zielsetzung genaue Ziele definiert, deren Erreichung nachfolgend nach eigener Einschätzung bewertet wird.

5.1 Bewertung der Zielerreichung

Das primäre Ziel ist erfüllt, das Tool bietet den Usern an, über eine einfache Maske Daten in das JSON-LD Format zu konvertieren.

Mit der ausführlichen Dokumentation des Projektes wurde zudem eine gute Grundlage für den Unterhalt und die Weiterentwicklung des Tools geschaffen.

Des Weiteren sind bereits viele Verbesserungen implementiert, die über das Primärziel hinausgehen. Alle Masken (inkl. die wichtigsten Tutorials für Endbenutzer) sind viersprachig vorhanden, resp. können via XML-Template viersprachig konfiguriert werden. Zusätzlich werden automatisiert und dynamisch die aktuellsten Definitionen gemäss schema.org live abgegriffen und maschinell in die User-Sprache übersetzt. Um die Qualität der XML-Templates zu verbessern und die Administration zu erleichtern, wurde eine hochwertige Validierungsfunktion erstellt, die nicht nur auf das XML-Schema prüft, sondern auch erweiterte Prüfungen mittels XPath Abfragen durchführt.

  • Intuitives webbasiertes Formular

    Das Formular ist einfach zu bedienen und orientiert sich optisch an gängigen Formularen auf Websites. Dies bietet Nutzern einen gewissen Wiedererkennungswert, was es intuitiv macht. Die Platzierung auf der Startseite macht es leicht auffindbar und Nutzer können ohne Umwege starten. Unterstrichene Begriffe beinhalten Beschreibungen, die automatisch aus schema.org übersetzt werden. Hier besteht allenfalls Optimierungsbedarf, da die Übersetzungen maschinell geschehen und die Sätze je nach dahinterstehendem HTML-Element nicht vollständig sind. Um die Qualität der übersetzen Elemente zu erhöhen, könnte zusätzlich eine manuelle Übersetzung von ausgewählten Elementen durchgeführt werden, es ist jedoch fraglich ob der dadurch entstehende Aufwand gerechtfertigt wäre.

  • Umfassendes Tutorial

    Das Tutorial für die Nutzung des Formulars ist kurz und verständlich gehalten. Es ist in einer einfachen Sprache verfasst und auf komplizierte Satzstellungen wurde bewusst verzichtet. Erweiterungspotential bietet sich besonders in der Erläuterung der JSON-LD-Einbindung, eine erste Empfehlung findet sich im nächsten Kapitel. Das Tutorial ist mit Screenshots eines konkreten Beispiels versehen, was es für Nutzer noch einfacher macht.

  • Anleitung zur Erstellung von XML-Templates

    Das XML-Tutorial für die Generierung neuer Formulare ist detailliert und dennoch kurz gehalten. Es legt den Aufbau eines XML-Files dar und beschreibt die Elemente und Attribute die vorhanden sein müssen und optional genutzt werden können. Zudem wurden gezielt XML-Templates erstellt, die als Beispiel dienen können: Ein komplexes Beispiel, ein einfaches Beispiel und jeweils Testfiles für alle Formate und Hierarchieausprägungen zu Testzwecken. 6 Erweiterungsideen und Empfehlungen

Back to top

Das entwickelte Tool hat Potential für Weiterentwicklungen. Erste Ideen und Empfehlungen werden nachfolgend aufgeführt. Dabei wird unterschieden zwischen funktionaler Weiterentwicklung und der Optimierung des Layouts.

6.1 Funktionale Erweiterungen

Die gesamte Website ist, soweit möglich, „responsive“ und damit Smartphone-freundlich aufgebaut. Dies mit dem Hintergedanken, dass beispielsweise Bibliotheksmitarbeitende das Formular auch dann ausfüllen können, wenn sie unterwegs bzw. nicht an einem stationären Computer sind (beispielsweise aber an einem Tablet-Computer). Damit der JSON-LD-Datensatz dennoch schnellstmöglich von den Administratoren in die Website eingepflegt werden kann, bietet es sich an, das Formular um einen E-Mail-Versand zu erweitern. Per Knopfdruck soll der Datensatz in eine E-Mail kopiert und versandt werden können, was jedoch erweiterte Konfigurationen bezüglich IT-Security (spam-mails / robos) und Mailserver bedingt.

Eine weitere Möglichkeit, den generierten Datensatz schnell verschicken oder in die Website einpflegen zu können, ist die Speicherung in der Zwischenablage. Mit einem Button „In Zwischenablage speichern“ kann dafür gesorgt werden, dass der gesamte Datensatz mit einem Klick zur Weiterverarbeitung gesichert wird. Dadurch werden den Bibliotheksmitarbeitenden die Schritte des Markierens sowie des Kopierens erspart.

Für weitere Zeitersparnis könnte eine Upload-Funktion integriert werden. Diese soll den Bibliotheksmitarbeitenden ermöglichen, alte Datensätze erneut ins Formular zu laden um dann beispielsweise nur geringfügige Änderungen im Formular vorzunehmen. Vergangene Events, die sich in den Folgejahren ähnlich wiederholen, könnten im JSON-LD-Format gesichert und zukünftig erneut, mit wenigen Anpassungen, verwendet werden. Die Erweiterung ist jedoch sehr Aufwändig, da eine „Rückkonvertierung“ von JSON-LD in XML nicht vorgesehen ist. Zudem sind Security-Aspekte zu berücksichtigen (beispielsweise das Hochladen von Schadcode verhindern). Als Workaround kann aktuell ein XML-Template dienen, dass Default-Werte enthält.

Mit den aktuellen XML-Vorlagen können die Formulare in den Sprachen Deutsch, Englisch, Französisch und Italienisch ausgegeben werden. Die Übersetzung erfolgt manuell im XML-File, indem die Labels für die Sprachen befüllt werden. Das bedeutet, dass die Formulare nicht in anderen Sprachen zur Verfügung stehen, wenn diese nicht bewusst übersetzt werden. Dennoch erscheinen die Sprachmöglichkeiten in der Dropdown-Liste auf der Formular-Seite. Um Unmut seitens der Endnutzer zu vermeiden, für die die gewünschte Sprache trotz Anzeige nicht zur Verfügung steht, empfiehlt sich eine dynamisch generierte Dropdown-Liste. Sprachen sollten demnach nur angezeigt werden, wenn die Labels entsprechend ausgefüllt wurden. Werden die Labels leer gelassen, soll die Sprache nicht in der Liste zur Verfügung stehen. Da der Aufwand verhältnismässig gross ist, beispielsweise man feststellt, dass eine Sprache nicht vergessen wurde, sondern absichtlich fehlt, ist diese Optimierung noch nicht erfolgt.

Das Tool kann nach heutigem Stand aus JSON-LD aus einem XML-Template erstellten. Zudem verarbeitet es live schema.org-Referenzen für die Übersetzungen im Formular. Hier bietet es sich als Weiterentwicklung an, die Generierung von XML-Templates zu automatisieren. Zeitgleich könnten schema.org-Klassen vorgeschlagen oder gegengeprüft werden. Dies würde die Bereitstellung von Formularen unterstützten und könnte der Anwendung eine breitere Nutzung ermöglichen.

6.2 Ideen für Design-Optimierungen

Um das User Interface zu optimieren, können noch einige weitere Massnamen und Erweiterungen vorgenommen werden.

Zum einen ist die Navigation noch nicht an allen Stellen gleichermassen konsistent, vor allem aber sollte sie mit zunehmendem Umfang der Anwendung noch etwas vereinfacht werden. Wohingegen zum jetzigen Zeitpunkt das Dropdown zur Auswahl der Formulare passend ist, kann diese Form der Navigation mit steigendem Umfang nicht mehr ausreichend sein. Dann empfiehlt es sich, eine linke Navigationsbox zu integrieren und die Formulare beispielsweise zu kategorisieren.

Bei der bisherigen Herangehensweise der Konvertierung der XML-Dateien zu HTML-Formularen werden keine Klassen berücksichtigt. Da auch die IDs der HTML Tags automatisch generiert werden, wird empfohlen das XML zu erweitern.

Sollte der Wunsch bestehen, die Tutorials zu erweitern und weitere Tutorials hinzuzufügen, so ist zum einen eine Vorlage im Ordner Tutorials zu finden (tutorials/template.php). Des Weiteren ist die Menüführung in der Datei tutorials/create.php bereits als HTML vorhanden und muss nur re-aktiviert werden (derzeit ist der Code auskommentiert).

6.3 Empfehlungen für die Dokumentation

Um CMS-Verantwortlichen die Einbindung des JSON-LD-Datensatzes in die Website zu erleichtern, wird eine ausführliche Erweiterung des Tutorials empfohlen. Insbesondere wenn nur begrenzt Kenntnisse zur Handhabung des CMS vorhanden sind, kann die Integration externer Links zu konkreten CMS-Tutorials hilfreich sein. Für das CMS Wordpress gibt es beispielsweise diverse Plug-Ins9, um den Header zu bearbeiten, die Anzahl solcher Erweiterungen stellt Verantwortliche aber auch vor die Qual der Wahl. Links mit guten und einfach einsetzbaren Plug-Ins können demnach auch die Motivation steigern.
